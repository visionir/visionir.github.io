细说String类型

查看源码可知，String类实现了java.io.Serializable, Comparable, CharSequence三个接口。

Serializable用于标识String类的可序列化

Comparable用于比较两个对象的大小

CharSequence是用于描述字符串结构的接口

private final char value[] 用于存储字符串且初始化后内容不能改变

private int hash 存储其hash值，默认为0，由于String是final类型，因此其hash不会频繁变更，确保了唯一性

private static final long serialVersionUID = -6849794470754667710L 用于标识序列化版本号，用于判定反序列化(指从文件中读取对象)时字节流与本地类的一致性，如果相同则认为一致，可以进行反序列化，如果不同抛出异常

private static final ObjectStreamField[] serialPersistentFields = new ObjectStreamField[0] 用于指定需要被序列化的字段，transient标识的字段以及static变量默认无法序列化，但serialPersistentFields 会覆盖transient的作用

为什么String类型被定义为final类型呢？

为了保证String类型的高效性以及安全性，由于被final修饰的类无法被继承，所以可以防止String类中的方法被重写，同时其所有方法都会成为被JVM认定是内联函数，因而提高性能。

关于字符串常量池的描述

在JDK1.7版本以前字符串常量池存放在方法区内，方法区位于永久(PermGen)代中与堆(Heap)隔离；而从JDK1.7开始存储在永久代的部分数据就已经转移到Java Heap或者本地内存(Native memory)中，但永久代仍存在并没有完全移除，譬如符号引用(Symbols)转移到了本地内存(Native memory)；字符串常量池(interned strings)、类的静态变量(class statics)转移到了(堆)heap中。对于字符创常量池中存放的是对象还是对象的引用众说纷纭，由于本人能力有限，请参考

前者JVM会首先检查常量池中是否存在该字面量：

  • 若存在，则直接返回此字面量的引用；
  • 若不存在，则在常量池中创建该字面量，返回其引用；

后者JVM同样会先检查常量池中是否存在对应的字面量：

  • 若存在，则在堆中创建String对象，在对象内部引用该字面量；
  • 若不存在，则先在常量池中创建字面量，然后在堆中创建String对象，在对象内部引用该字面量；

结论是后者会创建1-2个对象，一个在堆中，一个在字符串常量池中；前者创建0-1个对象，在字符串常量池中。

我更倾向字符串常量池中存储的是字符串对象的引用的说法(RednaxelaFX作者的分析最透彻)，对于String s=""; String s=new String();创建了几个对象的问题其实有许多歧义，可以参考上面的博客进行分析。

有关字符串的算法题

给定一个主串和一个模式串，找出模式串在主串中的位置。

此类字符串的模式匹配问题,常用KMP算法解决.

KMP算法由D.E.Knuth，J.H.Morris和V.R.Pratt提出的，因此人们称它为克努特—莫里斯—普拉特操作（简称KMP算法）。KMP算法的核心是利用匹配失败后的信息，尽量减少模式串与主串的匹配次数以达到快速匹配的目的。具体实现就是通过一个next()函数实现，函数本身包含了模式串的局部匹配信息。KMP算法的时间复杂度O(m+n)。
